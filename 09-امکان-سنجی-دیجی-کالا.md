# امکان‌سنجی: دیجی‌کالا و سایت‌های واقعی

---

## خلاصه یافته‌ها

<div dir="rtl">

X-WebAgentBench یک محیط شبیه‌سازی‌شده‌ی بسته (WebShop) است، نه ابزاری برای سایت‌های واقعی. سه بخش اصلی آن بررسی شد تا مشخص شود کدام بخش برای سناریوی سایت واقعی (بدون ترجمه، صفحه شلوغ) قابل استفاده است.

</div>

---

## تحلیل سه بخش کلیدی

### ۱. منطق امتیازدهی (reward) — `goal.py:230-272`

<div dir="rtl">

فرمول امتیاز کاملاً مخصوص خرید محصول است:

</div>

```
reward = (تعداد attribute های منطبق + تعداد option های منطبق + قیمت_درست)
         ─────────────────────────────────────────────────────────────  × r_type
              (تعداد کل attributes + تعداد کل options + 1)
```

<div dir="rtl">

این فرمول به **فیلدهای ساختاریافته و برچسب‌خورده** وابسته است که در سناریوی دیجی‌کالا وجود ندارند:
- `goal['attributes']`, `goal['goal_options']`, `goal['price_upper']` — از ground-truth محصول
- محصول خریداری‌شده باید `Attributes`, `Title`, `BulletPoints`, `Description`, `product_category` داشته باشد
- مطابقت با thefuzz (آستانه ۸۵) و spaCy (تطبیق اسم‌ها) و `normalize_color`

**نتیجه:** برای صفحه‌ای پر از تبلیغ و آفر، این متریک هیچ سیگنالی نمی‌دهد.

</div>

### ۲. کد تولید دیتاست — `generate_attrs.py`

<div dir="rtl">

از TF-IDF روی `name + small_description` (گروه‌بندی‌شده با category) ویژگی استخراج می‌کند. یعنی:
- ورودی باید کاتالوگ تمیز و ساختاریافته باشد (نام، توضیح کوتاه، دسته)
- دستورالعمل‌های انسانی (`items_human_ins`) جداگانه با MTurk جمع شده‌اند، نه با این کد

**نتیجه:** صفحه‌ی خام یک سایت واقعی مستقیماً به این pipeline خورانده نمی‌شود.

</div>

### ۳. موتور محیط و observation — `WebShopEnv.py:22-106`

```python
if t.parent.name == 'button':          processed_t = f'\n[{t}] '
elif t.parent.name == 'label':         # options
elif t.parent.get('class') == ["product-link"]:  # asins
```

<div dir="rtl">

این فقط روی HTML تمیز و templated خود سرور Flask کار می‌کند. روی DOM واقعی یک سایت با تبلیغ و کادر، این پارسر **کاملاً می‌شکند**.

</div>

## ارزیابی خودکار: چطور کار می‌کند (جریان کامل)

### پیش‌نیاز: ساختار سه‌لایه داده

```
items_shuffle/       ← محصولات (40k محصول با تمام جزئیات)
     ↓
items_ins/           ← ویژگی‌های استخراج‌شده (TF-IDF) برای هر محصول
     ↓
items_human_ins/     ← دستورالعمل انسانی + هدف دقیق برای 200 محصول
```

### جریان ارزیابی

```
[items_human_ins] → هدف تعریف می‌شود
        ↓
عامل دستورالعمل را می‌بیند ("i need a 9.5 hiking shoe with rubber sole")
        ↓
عامل در محیط Flask جستجو/کلیک می‌کند
        ↓
عامل click[Buy Now] می‌زند
        ↓
محصول خریداری‌شده vs هدف → get_reward() محاسبه می‌شود
        ↓
امتیاز بین 0.0 تا 1.0
```

### قلب سیستم: `get_reward()` در `goal.py:230`

```python
total_reward = (
    (num_attr_matches + num_option_matches + r_price)
    / (len(goal['attributes']) + len(goal['goal_options']) + 1)
)
total_reward *= r_type  # جریمه اگر نوع محصول کاملاً اشتباه باشد
```

## چرا برای دیجی‌کالا کار نمی‌کند؟

### مشکل اول — `instruction_attributes` از کجا می‌آید؟

<div dir="rtl">

این فیلد را کارگران MTurk آمازون برای هر محصول دستی نوشتند. برای دیجی‌کالا باید یکی از اینها را انجام داد:
- الف) دوباره با انسان این کار را کنی (گران)
- ب) با LLM از توضیحات محصول ویژگی استخراج کنی (قابل اتوماسیون)

</div>

### مشکل دوم — Attributes محصول از کجا می‌آید؟

<div dir="rtl">

در `engine.py:371-374` از فایل `items_ins/` خوانده می‌شود که با TF-IDF از متن محصول استخراج شده. برای دیجی‌کالا باید همین pipeline را اجرا کنی.

</div>

### مشکل سوم — مقایسه متنی به فارسی

```python
# goal.py:199-203:
g_attr in purchased_product['Title'].lower()
g_attr in ' '.join(purchased_product['BulletPoints']).lower()
```

<div dir="rtl">

این string matching ساده برای فارسی کار می‌کند — اما `lower()` فارسی بی‌معنی است. مشکل اصلی‌تر `r_type` است که از **spaCy انگلیسی** استفاده می‌کند.

</div>

## راه‌حل عملی برای دیجی‌کالا

```python
# مرحله ۱ — crawl محصولات دیجی‌کالا → JSON با همان ساختار

# مرحله ۲ — تولید items_ins/ (ویژگی‌های هر محصول):
# به جای TF-IDF، می‌توانی از LLM استفاده کنی:
prompt = f"ویژگی‌های کلیدی این محصول را استخراج کن:\n{product_description}"
# خروجی: ["نویزکنسلینگ", "بلوتوث ۵.۰", "باتری ۳۰ ساعته"]

# مرحله ۳ — تولید items_human_ins/ (دستورالعمل‌ها):
# یا انسانی یا با LLM:
prompt = f"یک دستورالعمل خرید برای این محصول بنویس:\n{product_name}\nویژگی‌ها: {attributes}"
# خروجی: {
#   "instruction": "دنبال هدفون بی‌سیم با نویزکنسلینگ هستم...",
#   "instruction_attributes": ["نویزکنسلینگ", "بلوتوث ۵.۰"],
#   "instruction_options": ["مشکی"]
# }

# مرحله ۴ — تغییر get_reward() برای فارسی:
# حذف spaCy انگلیسی از r_type یا جایگزینی با hazm
# r_attr و r_price نیاز به تغییر زیادی ندارند
```

<div dir="rtl">

**نتیجه:** قسمت ارزیابی خودکار این محیط کاملاً **داده‌محور** است — نه منطق پیچیده‌ای دارد. اگر داده‌ها را به فرمت درست آماده کنی، بقیه سیستم با تغییرات کمی کار می‌کند.

</div>

## چه چیزی از X-WebAgentBench به دردتان می‌خورد؟

<div dir="rtl">

**قابل اقتباس:**
- **فلسفه‌ی متریک reference-based:** تعریف goal به‌صورت «محصول مرجع + مجموعه ویژگی»، و امتیاز = درصد همپوشانی ویژگی‌ها به‌جای قضاوت انسانی
- ساختار حلقه‌ی ReAct و قالب‌های prompt (`model.py`, `WebShop_prompt.py`) — مستقل از اینکه محیط چیست
- جداسازی reward به زیرمؤلفه‌ها برای تحلیل خطا (چرا عامل شکست خورد: type؟ attribute؟ option؟)

**باید از صفر بسازید:**
- استخراج observation از DOM واقعی (یا accessibility tree / screenshot) — کاملاً جدید
- تعریف ground-truth و منطق success برای سایت واقعی
- مدیریت عدم‌قطعیت و content drift سایت زنده (تکرارپذیری؛ مشکل اصلی همه‌ی بنچمارک‌های سایت واقعی)

</div>

## پیشنهاد: بنچمارک‌های جایگزین برای سایت واقعی

<div dir="rtl">

برای سناریوی سایت واقعی (DOM شلوغ، تبلیغ، آفر)، نسل دیگری از بنچمارک‌ها طراحی شده‌اند:

| بنچمارک | توضیح | مزیت |
|---------|-------|------|
| **WebArena** | کپی reproducible از سایت‌های واقعی (با همان پیچیدگی DOM) | تکرارپذیری بالا، reward قطعی |
| **WebVoyager** | تسک روی سایت‌های زنده‌ی واقعی + ارزیاب خودکار مبتنی بر MLLM | برای صفحات شلوغ واقعی، بدون نیاز به برچسب‌گذاری ساختاریافته |
| **Mind2Web** | ۲٬۰۰۰ تسک از ۱۳۷ سایت واقعی | متریک‌های element accuracy / operation F1 / step success rate |
| **BrowserGym** | اکوسیستمی که همه محیط‌ها را یکپارچه می‌کند | ترکیب چند بنچمارک |

**برای امتیازدهی روی صفحات واقعی شلوغ:** رویکرد WebVoyager (ارزیاب MLLM) عملی‌تر از فرمول ویژگی‌محور X-WebAgentBench است، چون نیازی به برچسب‌گذاری ساختاریافته‌ی هر صفحه ندارد.

</div>
